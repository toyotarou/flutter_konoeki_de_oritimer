# 降りタイマー (oriTimer) 基本設計書

> **基本設計書とは？**
>
> 基本設計書は「何を作るか」を定義するドキュメントです。
> 詳細設計書（どう作るか）の **前段階** にあたります。
>
> **基本設計書と詳細設計書の違い:**
>
> | 観点 | 基本設計書 | 詳細設計書 |
> |---|---|---|
> | 読む人 | 発注者・PM・開発者 | 開発者 |
> | 書く内容 | 何を作るか（要件・機能） | どう作るか（実装方法） |
> | 技術の深さ | 技術に依存しない記述が理想 | 具体的な型名・関数名まで記載 |
> | 例 | 「駅に近づいたら通知する」 | 「NativeGeofenceManagerでentryイベントを500m半径で登録する」 |
>
> 基本設計書は以下の観点で構成されます:
>
> | セクション | 書く内容 | なぜ必要か |
> |---|---|---|
> | 1. プロジェクト概要 | 背景・目的・スコープ | 「なぜこのアプリを作るのか」を共有するため |
> | 2. システム構成 | アプリとOSサービスの関係 | 全体の登場人物を明確にするため |
> | 3. 機能一覧 | アプリが持つ全機能のリスト | 「何ができて何ができないか」を合意するため |
> | 4. 機能要件 | 各機能の振る舞い | 開発者の「こう解釈した」を防ぐため |
> | 5. 画面一覧・遷移 | 画面の構成と流れ | ユーザー体験を設計するため |
> | 6. データ設計 | 扱うデータの種類と関係 | APIとUIの整合性を保つため |
> | 7. 外部インターフェース | API仕様の概要 | サーバーとの約束を定義するため |
> | 8. 非機能要件 | 性能・セキュリティ等 | 「動くけど使えない」を防ぐため |

## 1. プロジェクト概要

> **このセクションの目的：** 「なぜこのアプリを作るのか」「誰が使うのか」「どこまでが範囲か」を明確にする。プロジェクトに関わる全員が最初に読むセクション。

### 1.1 背景

電車通勤・通学時に居眠りで目的の駅を乗り越してしまう問題は、時刻指定アラームでは根本的に解決しにくい。電車の遅延や乗り換えが多いルートでは、目的駅の到着時刻を事前に正確に把握することが難しいためである。

本アプリは、スマートフォンのジオフェンス機能を活用し、「時刻」ではなく「場所（駅）」をトリガーとした降車通知を実現する。個人学習を兼ねて、Flutter + ネイティブジオフェンスAPIの構成で開発する。

### 1.2 目的

- 選択した東京の駅の半径1000m圏内に入ったとき、プッシュ通知・ループバイブレーションでユーザーに通知する
- 時刻に依存せず、場所ベースで確実に乗り越しを防ぐ
- Flutter / ジオフェンスAPI / Riverpod の学習教材としても活用する

### 1.3 対象ユーザー

- 開発者自身（個人利用）
- 電車通勤・通学中に乗り越しが心配な知人（将来的な共有を想定）

### 1.4 スコープ（対象範囲）

> **補足：** スコープとは「このプロジェクトで作るもの・作らないもの」の境界線。これを書いておかないと、「あれも欲しい、これも欲しい」と際限なく膨らんでしまう（スコープクリープと呼ぶ）。

**対象範囲（作るもの）:**

- 東京の主要路線・駅の一覧表示（APIから取得）
- 降りたい駅の選択機能
- 選択駅のジオフェンス監視の開始・停止
- 駅圏内進入時のプッシュ通知（バイブレーション付き）
- 位置情報・通知パーミッションの要求
- 監視状態・選択駅の画面表示

**対象外（作らないもの）:**

- 東京以外の路線・駅への対応
- 複数駅の同時監視
- オフライン対応（駅データのローカルキャッシュ）
- 乗車履歴の記録
- 地図表示
- 時刻表の表示

│  │  - 通知発火ロジック     │      │  - 境界線の監視           │  │

### 2.2 各コンポーネントの役割

| コンポーネント | 技術 | 役割 |
|---|---|---|
| Flutterアプリ | Flutter (Dart) | ユーザーインターフェース。駅選択・監視開始・通知発火を担当 |
| OSジオフェンスサービス | iOS: CoreLocation / Android: Geofence API | 端末の位置情報を監視し、境界線の通過イベントを発火 |
| 外部APIサーバー | toyohide.work | 東京の路線・駅データをJSON形式で提供 |

### 2.3 通信方式

| 項目 | 内容 |
|---|---|
| プロトコル | HTTP |
| データ形式 | JSON |
| 通信方向 | クライアント → サーバー（リクエスト/レスポンス型） |
| 通信方法 | HTTP POST |
| 認証 | なし |

### 2.4 ジオフェンスの仕組み

> **補足：** ジオフェンスはネイティブのOS機能を利用するため、Flutterアプリが終了した状態でも動作する。OSがジオフェンス境界の通過を検知し、登録済みのコールバック関数を呼び出す。

```
1. Flutterアプリがジオフェンスを登録（駅の座標 + 半径1000m）
            │
            ▼
2. OSがバックグラウンドで位置情報を監視
            │
            ▼
3. デバイスが駅の1000m圏内に入る
            │
            ▼
4. OSがコールバック関数を呼び出す
            │
            ▼
5. コールバック内でプッシュ通知を発火する
```

| F-06 | ジオフェンス監視停止 | 登録済みのジオフェンスを全削除し、監視とバイブレーションを停止する | 必須 |

## 4. 機能要件

> **このセクションの目的：** 各機能の「振る舞い」を自然言語で定義する。技術的な実装方法は書かず、「ユーザーから見てどう動くか」を記述する。

### F-01 駅データ取得

```
概要:    アプリ起動時に外部APIから全路線・駅データを取得する
入力:    なし（自動取得）
処理:    外部APIサーバーへPOSTリクエストを送る
出力:    路線名・駅名・緯度経度のリスト
```

**異常系:**
- 通信エラー時はエラーメッセージを表示する
- データ取得中はローディング表示をする（ただし本実装では省略）

### F-02 路線・駅一覧表示

```
概要:   取得した駅データを路線ごとにグループ化して一覧表示する
入力:   F-01で取得した駅データ
処理:   路線名をグループヘッダーとし、各駅を配下にリスト表示する
出力:   路線グループ化された駅リスト
```

**表示項目（各駅）:**
- 選択インジケーター（選択中は黄色の丸、非選択は白枠の丸）
- 駅名
- 緯度・経度

**表示ルール:**
- 路線名をヘッダーとして表示する
- データは取得順（API提供順）に表示する

### F-03 駅選択

```
概要:   路線・駅一覧から降りたい駅を1つ選択する
入力:   駅アイテムのタップ
処理:   選択状態をアプリ内で保持する
出力:   選択駅名の画面表示更新、インジケーターの強調表示
```

**動作ルール:**
- 選択できる駅は1つのみ（複数選択不可）
- 別の駅をタップすると選択が切り替わる
- 選択した駅名は画面上部の「現在の選択」欄に表示される

### F-04 ジオフェンス監視開始

```
概要:   選択した駅の座標にジオフェンスを登録し、監視を開始する
入力:   監視開始ボタン（目のアイコン）のタップ
処理:   選択駅の緯度経度を中心に、半径1000mのジオフェンスを登録する
出力:   監視中状態への遷移（ボタンが黄色に変わる）
```

**動作ルール:**
- ジオフェンスのトリガーは「入場（圏内への進入）」のみ
- 監視開始前に位置情報パーミッションが付与されていることを確認する
- パーミッションが未取得の場合は、パーミッション要求を促すメッセージを表示する

### F-05 降車通知

```
概要:   ジオフェンス圏内に入ったとき、音量最大化・ループバイブレーション・プッシュ通知で知らせる
入力:   OSからのジオフェンス入場イベント（バックグラウンド動作）
処理:   音楽ストリームの音量を最大にし、通知を発火し、バイブレーションをユーザーが停止するまでループさせる
出力:   音量最大化（Android のみ・音楽ストリーム）
        プッシュ通知（タイトル「降りる駅アラーム」、ボディに駅名を表示）
        バイブレーション（ユーザーが停止操作するまで繰り返し）
```

**動作ルール:**
- アプリがバックグラウンドまたは終了状態でも動作する
- 音楽ストリームの音量を最大（1.0）に引き上げる（Android のみ）。システムUIの音量スライダーは表示しない
- 音量の復元は自動では行われない。ユーザーが手動で調整する
- バイブレーションは最大強度（Android: intensity 255）でパターンをループし続ける
- バイブレーションの停止はユーザーがアプリの停止ボタン（×）を押したときのみ行われる
- 通知タイトル: 「降りる駅アラーム」
- 通知本文: 対象駅名を含む

### F-06 ジオフェンス監視停止

```
概要:   登録済みの全ジオフェンスを削除し、監視とバイブレーションを停止する
入力:   停止ボタン（×アイコン）のタップ
処理:   OSに登録済みのジオフェンスを全件削除し、ループ中のバイブレーションも即時停止する
出力:   非監視状態への遷移（目のアイコンが通常色に戻る）
```

**動作ルール:**
- 停止後は、圏内に入っても通知は発火しない
- ループ中のバイブレーションは停止ボタンを押した瞬間に確実に止まる
- 選択駅の情報はリセットされる（画面の選択表示が「(未選択)」に戻り、SharedPreferencesからも削除される）

### F-07 パーミッション要求

```
概要:   ジオフェンスと通知の動作に必要なパーミッションをユーザーに要求する
入力:   パーミッション要求ボタン（錠前アイコン）のタップ
処理:   位置情報（Fine位置情報）→ 位置情報（常に許可）→ 通知の順に要求する
出力:   OSのパーミッションダイアログ表示、付与状況のアイコン更新
```

**動作ルール:**
- Fine位置情報 → 常に許可 → 通知の順に順次要求する
- 既に付与済みのパーミッションはスキップする
- 「常に許可」はOSの設定画面に誘導が必要な場合がある

### F-08 監視状態表示

```
概要:   現在の選択駅名と監視中かどうかの状態を画面に表示する
表示内容:
  - 選択駅名（未選択時は「(未選択)」）
  - 監視中は「監視中」テキストを表示、停止中は非表示
```

### F-09 パーミッション状態表示

```
概要:   パーミッションが付与済みかどうかをAppBarのアイコン色で示す
表示ルール:
  - パーミッション付与済み: 錠前アイコンが黄色
  - 未取得: 錠前アイコンが通常色（白）
  - 監視中: 目のアイコンが黄色
  - 停止中: 目のアイコンが通常色（白）
```

### F-10 状態の永続化

```
概要:   アプリを再起動しても監視中状態（目のアイコン）と選択駅の両方を復元する
入力:   アプリ起動
処理:   起動時に SharedPreferences から監視状態・選択駅を読み込み、画面に反映する
出力:   前回終了時の監視状態と選択駅名がそのまま表示される
```

**動作ルール:**

| タイミング | 監視状態（isSetStation） | 選択駅（selectedStation） |
|---|---|---|
| 目のアイコンをタップ | `true` を保存 | 変更なし（既に保存済み） |
| 駅をタップして選択 | 変更なし | 選択駅の JSON を保存 |
| 停止ボタン（×）をタップ | 削除 | **削除**（画面も「(未選択)」に戻る） |
| アプリ起動 | 読み込んで復元 | 読み込んで復元 |

- 注意: ジオフェンスの監視自体はOSが管理するため、アプリ再起動後も継続している。アイコンの状態だけがアプリ内で復元される

## 6. データ設計

> **このセクションの目的：** アプリが扱う「データ」の種類と関係を定義する。詳細設計書ではフィールドの型まで書くが、基本設計書では「どんなデータがあるか」「データ同士がどう関係するか」を概念レベルで記述する。

### 6.1 データの種類

| データ名 | 概要 | 主な項目 |
|---|---|---|
| 路線情報 | 東京の鉄道路線の基本情報 | 路線番号、路線名、所属駅リスト |
| 駅情報 | 各駅の詳細情報 | 駅ID、駅名、住所、緯度、経度 |
| アプリ状態 | アプリの現在の動作状態 | ジオフェンス監視中かどうか、選択駅情報（どちらも永続化あり） |

### 6.2 データの関係

```
┌──────────────────┐       ┌──────────────────┐
│    路線情報        │       │    駅情報          │
│                  │ 1   N │                  │
│  - 路線番号       │───────│  - 駅ID           │
│  - 路線名        │       │  - 駅名           │
│                  │       │  - 住所           │
│                  │       │  - 緯度           │
│                  │       │  - 経度           │
└──────────────────┘       └──────────────────┘
                                    │
                                    │ ユーザーが1つ選択
                                    ▼
                           ┌──────────────────┐
                           │   選択駅          │
                           │（アプリ状態）      │
                           │  - 選択中の駅情報  │
                           │  - 監視中フラグ    │
                           └──────────────────┘
```

**関係の説明:**
- 1つの路線に対して、N個の駅がある（1対多の関係）
- ユーザーが選択できる駅は任意の1駅のみ

### 6.3 データの取得タイミング

| データ | 取得タイミング | 保持方法 |
|---|---|---|
| 路線・駅情報 | アプリ起動時に全件取得（keepAlive） | Riverpod Providerでキャッシュ |
| 選択駅情報 | ユーザーの駅タップ時に更新 | Riverpod Notifierで管理 |
| 監視状態 | ジオフェンス開始・停止操作時に更新 | Riverpod Notifier + SharedPreferences（永続化）|
| 選択駅情報 | 駅タップ時に保存、停止ボタンで削除 | SharedPreferences（JSON文字列で永続化）|

> **補足：** `SharedPreferences` とは、アプリを終了しても消えない小さなデータを端末に保存する仕組み。Androidでは `XML` ファイル、iOSでは `NSUserDefaults` が裏側で使われる。監視状態の `true/false` のような小さなフラグの保存に適している。

> **補足：** 「keepAlive」とは一度取得したデータをアプリのライフサイクル全体を通して保持し続ける設定。駅データは頻繁に変わらないため、再取得せずに使い回す。

## 8. 非機能要件

> **このセクションの目的：** 「動くこと」以外の品質要件を定義する。機能要件が「何ができるか」なら、非機能要件は「どれくらいの品質で動くか」を定義するもの。

### 8.1 性能要件

| 項目 | 要件 | 備考 |
|---|---|---|
| 初期表示 | 3秒以内に路線・駅一覧を表示 | ネットワーク環境に依存 |
| ジオフェンス登録 | タップから2秒以内に登録完了 | ネイティブAPIの応答に依存 |
| 通知発火 | 圏内進入後、できる限り短時間で発火 | OSや省電力設定に依存するため厳密な保証は困難 |

### 8.2 可用性要件

| 項目 | 要件 |
|---|---|
| バックグラウンド動作 | アプリが終了状態でもジオフェンス監視・通知が機能すること |
| サーバー停止時 | エラーメッセージを表示。アプリはクラッシュしないこと |
| パーミッション未取得時 | 適切なガイダンスを表示し、アプリが停止しないこと |

### 8.3 ユーザビリティ要件

| 項目 | 要件 |
|---|---|
| 操作性 | 駅選択から監視開始まで3タップ以内で完了できること |
| 状態確認 | 監視状態・選択駅が画面一目で確認できること |
| パーミッション | パーミッション未取得時に、分かりやすいガイダンスを表示すること |
| テーマ | ダークテーマを採用する |
| バイブレーション | 到着時のバイブレーションはユーザーが停止操作するまでループし続けること |
| 状態復元 | アプリ再起動後も監視状態（目のアイコン）が前回の状態を維持していること |

### 8.4 保守性要件

| 項目 | 要件 |
|---|---|
| API接続先 | 1箇所の設定（`path.dart`）で変更可能 |
| データモデル | freezedによるコード生成で型安全性を担保 |
| 状態管理 | Riverpodで統一。Widgetから直接API通信しない |
| ファイル構成 | 役割別にディレクトリを分割（model / controllers / screens） |

### 8.5 セキュリティ要件

| 項目 | 現状 | 将来的な対応案 |
|---|---|---|
| 通信暗号化 | HTTP（非暗号化） | HTTPS化を推奨 |
| 位置情報の扱い | 通知発火のみに使用。外部には送信しない | 変更なし |
| 認証 | なし | 必要に応じてAPI Key等を導入 |

## 9. 用語集

> **このセクションの目的：** ドメイン特有の用語や、プロジェクト内での独自の言葉遣いを定義する。

| 用語 | 意味 |
|---|---|
| ジオフェンス | 地図上に設定した仮想的な境界線。境界線の通過時にイベントが発火する |
| 圏内進入 | ジオフェンスで設定した円の内側にデバイスが入ること（エントリーイベント） |
| バックグラウンド動作 | アプリが画面に表示されていない・終了している状態でも機能する動作 |
| パーミッション | アプリがOS機能（位置情報・通知）を使うためのユーザー許可 |
| 常に許可 | 位置情報パーミッションの一種。アプリがバックグラウンドでも位置情報を取得できる |
| ジオフェンスサービス | iOSのCoreLocation、AndroidのGeofencing APIなど、OSが提供するジオフェンス機能 |
| keepAlive | Riverpod Providerの設定。アプリのライフサイクル全体でデータを保持し続ける |
| Provider | Riverpodの仕組み。データの取得・保持・配信を担当する部品 |
| freezed | Dartのコード生成ライブラリ。イミュータブル（変更不可）なモデルクラスを自動生成する |
